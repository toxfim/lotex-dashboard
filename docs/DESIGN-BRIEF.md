## Блок A — Контекст и дизайн-система (вставить ПЕРВЫМ)

Ты — продуктовый дизайнер. Мы редизайним внутренний веб-дашборд **Lotex** —
инструмент для команды, которая автоматически участвует в тендерах на площадке
UzEx (xarid.uzex.uz) и продаёт товары в электронном магазине UzEx.

**Кто пользователь.** Оператор внутри компании. Работает за десктопом весь день,
обрабатывает поток тендерных лотов: система с помощью AI подбирает наш товар под
лот, оператор принимает или отклоняет, затем участвует в тендере. Плюс ведёт
каталог магазина, поставщиков (прайс-листы в Excel), сборку ПК под лоты и смотрит
аналитику. Это профессиональный рабочий инструмент, не маркетинговый сайт: данных
много, оператор возвращается сюда десятки раз в день, скорость и ясность важнее
«вау-эффекта».

**Язык интерфейса — русский.** Весь текст, подписи, кнопки, пустые состояния — на
русском. Числа, цены, ИНН, коды — моноширинным шрифтом с табличными цифрами.

**Направление стиля — чистый современный SaaS** в духе Linear, Vercel и Stripe:

- Много воздуха, спокойный нейтральный фон, тонкие границы, мягкие ненавязчивые тени.
- Один акцентный цвет, используется скупо — только для главного действия и активных
  состояний. Не «раскрашивать» интерфейс.
- Чёткая типографическая иерархия вместо рамок и жирных заливок.
- Плотность — «комфортно-компактная»: данных много, но между блоками есть дыхание.

**Две проблемы, которые редизайн ОБЯЗАН решить:**

1. **Непоследовательность.** Сейчас каждый экран выглядит по-своему: несколько
   разных систем кнопок, разные таблицы, четыре разных вида модалок/дровверов,
   верхняя панель переизобретена на каждой странице. Нужна **одна** система
   компонентов, применяемая везде одинаково.
2. **Непонятная иерархия.** На экране не видно, что главное; кнопки-действия
   теряются; статусы (принято/отклонено/открыт/закрыт/выгодно/невыгодно) читаются
   с трудом. Нужно, чтобы с первого взгляда было ясно: где заголовок, где главное
   действие, какой статус у объекта, куда смотреть.

### Дизайн-система (единая для всех экранов)

**Цвет.**

- Фон приложения — очень светлый нейтральный (почти белый, чуть тёплый/холодный —
  один тон). Поверхности карточек — белые. Границы — светло-серые, тонкие (1px).
- Текст: основной — почти чёрный; вторичный — средне-серый; третичный/подписи —
  светло-серый. Три уровня, чётко различимые.
- **Один акцент — синий** (ориентир на текущий бренд: `#2743E0`). Только главное
  действие, активный пункт меню, ссылки, выделение. Плюс мягкая версия акцента для
  фонов-подсветок.
- Семантические цвета, каждый в трёх видах (насыщенный / мягкий фон / текст):
  зелёный = успех/принято/выгодно, красный = ошибка/отклонено/невыгодно,
  янтарный = внимание/на грани/дедлайн скоро. Использовать ТОЛЬКО для статусов,
  не для декора.

**Типографика.** Гротеск для интерфейса (напр. Inter или IBM Plex Sans),
моноширинный для чисел/цен/кодов/ID (IBM Plex Mono, `tabular-nums`). Строгая шкала
размеров: заголовок страницы, заголовок секции, тело, подпись, микро-лейбл (капсом
с трекингом — использовать редко).

**Сетка и отступы.** Шкала на базе 4px. Радиусы: карточки ~12–14px, мелкие
элементы ~8px, чипы/пилюли — полукруглые. Тени — мягкие и лёгкие, максимум два
уровня (карточка / всплывающий слой). Никаких жёстких чёрных теней.

**Единые компоненты — спроектируй их один раз, дальше переиспользуй везде:**

- **Кнопки:** ровно один набор с вариантами — primary (акцент, одна на экран),
  secondary (контур), ghost (без фона), danger (красный), плюс icon-button. Один
  размер по умолчанию + компактный. Никаких параллельных систем кнопок.
- **Верхняя панель страницы (page header):** единый паттерн для ВСЕХ страниц —
  слева заголовок + подзаголовок, справа поиск/фильтры и главное действие. Одинаковая
  высота и отступы везде.
- **Таблица данных:** один компонент — «зебра» не нужна, разделять тонкими
  границами; заголовки — микро-лейблы; числовые колонки выравнивать вправо, моно;
  вся строка кликабельна (открывает деталь); сортируемые колонки со стрелкой; hover
  подсвечивает строку.
- **Статус-пилюля:** единый компонент, цвет по семантике (принято=зелёный,
  отклонено=красный, ожидает=нейтральный, открыт=акцент/нейтр., закрыт=серый,
  выгодно/на грани/невыгодно = зелёный/янтарный/красный). Мягкий фон + цветной текст,
  без кричащих заливок.
- **Оверлеи:** одна система. Правый **дровер** — для деталей объекта (лот, кредо).
  Центральная **модалка** — для форм/мастеров. Маленький **диалог** — для
  подтверждений. Не смешивать четыре разных.
- **Чип матча (match ring/score):** процент совпадения товара с лотом (0–100),
  цвет по порогам (высокий=зелёный, средний=янтарный, низкий=красный).
- **Пустое / загрузка / ошибка:** единые состояния для всех списков и таблиц —
  иконка + заголовок + пояснение + (опц.) действие. Загрузка — скелетоны, не
  «крутилка» с текстом. Ошибка сервера — дружелюбный экран с кнопкой «Повторить».

**Доступность и адаптивность.** Контраст AA. У icon-only кнопок — подписи/тултипы.
Фокус виден. Раскладка рассчитана на десктоп (главный сценарий), но не ломается на
узком ноутбуке: двухпанельные экраны схлопываются в одну колонку, таблицы получают
горизонтальный скролл внутри своей области, а не растягивают страницу.

## Блок B — Глобальный каркас (вставить вторым, до экранов)

Спроектируй оболочку приложения:

- **Левый сайдбар** (фикс. ширина, ~240px). Сверху бренд «Lotex» с меткой
  «Tender Matching». Секция **«РАБОЧАЯ ОБЛАСТЬ»**: Очередь (иконка «входящие»,
  с бейджем-счётчиком ожидающих), Все лоты, Электронный магазин, Поставки,
  Сборка ПК, Аналитика. Секция **«СИСТЕМА»**: Аккаунты, Настройки. Внизу —
  **карточка статуса AI-конвейера** (живой индикатор: активен/ожидает, «Просканировано
  N», «Сопоставлено M», «Последняя синхр. — X мин назад»). В самом низу — чип
  пользователя: инициалы-аватар, имя, логин, кнопка выхода.
- **Активный пункт меню** ясно выделен акцентом (заливка-подсветка + цветная
  иконка/текст), остальные — нейтральные.
- Контент справа: единый **page header** (см. дизайн-систему) + область страницы.
- Тёмной темы пока нет — только светлая, но токены цвета задай так, будто тёмную
  добавим позже (семантические переменные, не хардкод хексов).

## Блок C — Экраны (просить ПО ОДНОМУ, ссылаясь на дизайн-систему)

> Порядок по важности. Начни с «Очереди» — это сердце продукта.

### 1. Очередь (`/queue`) — ГЛАВНЫЙ экран

Рабочий стол оператора: поток AI-подобранных лотов, которые ждут решения
«принять / отклонить».

- **Header:** заголовок «Очередь», подзаголовок «Тендерные лоты, подходящие
  компании». Справа: поиск («Лот, заказчик или ID»), кнопка **secondary** «Запустить
  сопоставление» (в процессе → «Идёт сопоставление…»), кнопка ghost «Пересопоставить»,
  колокол уведомлений, меню настроек.
- **Двухпанельная раскладка.** Слева — **список лотов**, справа — **панель деталей**.
- Над списком: **сегмент-табы** «Подходящие» / «Принятые» / «Отклонённые», у каждого
  счётчик. Строка «N лотов» + **понятный селектор сортировки** (по совпадению /
  по сроку / по стоимости) — именно выпадающий селектор с подписью, НЕ кнопка-циклер.
- **Карточка лота в списке:** миниатюра товара, категория, название, **кольцо
  совпадения %**, заказчик, чипы: макс. цена (сум), выгодность (выгодно/на грани/
  невыгодно), срок до дедлайна (или «Закрыт»).
- **Панель деталей** (справа): категория-чип + № лота + внешняя ссылка «Открыть на
  UzEx»; крупный заголовок; заказчик и регион; чипы «Рекомендованные юр. лица»;
  кольцо совпадения; изображение товара (клик → лайтбокс); мета-блоки (Количество,
  Макс. цена, Срок — со срочной подсветкой если дедлайн близко); три секции —
  **«Анализ совпадения»** (двухэтапный AI-конвейер: векторное сходство → проверка
  LLM, покажи как два шага), **«Технические характеристики»** (таблица), **«Цена и
  выгодность»**.
- **Панель действий внизу деталей:** primary «Участвовать» (открывает мастер ставки),
  и для ожидающих — «Отклонить» / «Принять». Если решение уже принято — полоска
  статуса с кнопкой «Вернуть в очередь» (undo). Подсказка о горячих клавишах
  A/R (принять/отклонить).
- **Иерархия:** самое заметное — кольцо совпадения, статус выгодности и кнопка
  «Участвовать». Дедлайн, если горит, — визуально тревожный.
- **Состояния:** скелетоны при загрузке; дружелюбный экран «Сервер недоступен» с
  «Повторить»; свои пустые состояния под каждый таб.

### 2. Все лоты (`/lots`)

Архивная таблица всех лотов.

- **Header** + три фильтра-селектора (Статус / Категория / Юр. лицо) + поиск + счётчик.
- **Таблица**, колонки: **Лот** (сортируемая; название + № + ссылка на UzEx) ·
  **Заказчик** (+ регион) · **Категория** (чип) · **Совпадение** (сортируемая;
  полоска + цветной %) · **Макс. цена** (право, моно) · **Срок** (дата / «Закрыт») ·
  **Статус** (пилюля). Клик по строке → **правый дровер** с деталями лота (тот же
  контент, что панель в Очереди) с действиями принять/отклонить/undo.
- **Пагинация:** пронумерованная (диапазон «1–12 / N», назад/вперёд). Используй
  ЭТОТ паттерн пагинации как единый для всех таблиц продукта.

### 3. Электронный магазин (`/shop`)

Наши предложения (офферы) на xarid.uzex.uz, сгруппированные по юр. лицу и статусу.

- **Header:** primary «Добавить товар» (→ мастер), настройки.
- **Полоса статусов:** чипы-пилюли по статусам оффера со счётчиками (по умолчанию
  активен «Опубликован»).
- Строка-мета «N юр. лиц · всего M объявлений»; далее **секция на каждое юр. лицо**
  (название + ИНН + счётчик) со своей таблицей: **Наименование** (+ марка/производитель/
  код) · **Категория** · **Цена** · **Кол-во** · **Платформа** (чипы Электронный/
  Национальный магазин) · **Статус**.
- Клик по строке → **модалка** с деталью оффера. Если статус = отклонён — блок
  **«Причина отклонения»** с критериями AI/модератора, процентами и текстом; сетка
  метаданных; ссылка «Смотреть на UzEx».

### 4. Добавить товар (`/shop/add`) — мастер из 3 шагов

Динамическая форма-реплика формы добавления оффера на UzEx; публикует товар как
офферы сразу по нескольким юр. лицам. Спроектируй **степпер** сверху (3 шага) с
единым стилем полей.

- **Шаг 1 «Выберите товар»:** чекбоксы платформ, селектор категории, поиск товара
  по каталогу (список результатов, radio-выбор; выбранное фиксируется с крестиком
  для сброса).
- **Шаг 2 «Заполните данные»:** карточки полей — Бренд + Производитель (комбо);
  Количество (степпер), Мин./Макс. партия, Цена за штуку, Итоговая сумма (вычисляемая,
  disabled); блок **«Условия»** (год выпуска, срок годности, гарантия, срок поставки,
  страна — с селекторами периодов); блок **«Технические параметры»** (динамические
  поля под товар); **«Регионы»** (регион → районы, мультивыбор); **«Изображения»**
  (до 10, первое — главное); **«Комментарий»** + тип публикации.
  **Важно про иерархию: покажи inline-валидацию под полями** (сейчас её нет — форма
  только гасит кнопку submit). У каждого обязательного поля — понятное состояние
  ошибки с текстом «почему нельзя отправить».
- **Шаг 3 «Выберите магазины»:** мастер-деталь — слева список юр. лиц (чекбоксы +
  «Выбрать все»), справа — финансовые счета выбранного юр. лица (radio, с балансами,
  бейджами основной/транзитный). Итог: «{n} магазинов · {m} счетов выбрано».
- **Липкая панель действий:** подсказка «Будет опубликовано в {n} магазинах»,
  «Отмена», primary «Опубликовать» (активна только когда форма валидна). Черновик
  автосохраняется.

### 5. Поставки (`/suppliers` и `/supplies`)

Поставщики и их товары (прайс-листы из Excel превращаются в переиспользуемые шаблоны).

- **Header:** «Запустить сопоставление», «Пересопоставить», primary «Добавить
  поставщика». Баннер со статистикой (поставщики / товары / загрузки) + пояснение
  про Excel-шаблоны.
- **Сегмент-табы** «Поставщики» / «Все товары» (со счётчиками).
- Таблица **поставщиков:** Поставщик (аватар + название + ИНН + заметка) · Контакты ·
  Товары (счётчик) · Загрузки · Действия (загрузить Excel / изменить / удалить).
- Таблица **товаров:** Наименование · Поставщик (ссылка) · Модель · Категория ·
  Excel (чип-код, скачивает исходник) · Себестоимость · Продажа. Фильтры + «Загрузить
  ещё».
- **Модалки:** добавить/изменить поставщика; **загрузка Excel** (drag-drop);
  **подтверждение маппинга** — AI-предложенное сопоставление колонок Excel, редактируемое,
  с живым превью 6 строк; диалог удаления.
  Эти таблица и модалки сейчас — главный источник непоследовательности (свои цвета,
  свои сегменты). Приведи их строго к общей дизайн-системе.

### 6. Сборка ПК (`/pc-build`)

Лоты на сборку ПК: конфигуратор совместимости и стоимости компонентов.

- **Header:** редактируемый курс USD→UZS (инпут + «Сохранить» + источник live/ручной +
  сброс к live), тумблер «Только выгодные», тумблер режима открытия (Панель / Окно),
  настройки.
- **4 KPI-карточки:** Лоты ПК / Собрано 100% / Выгодные / Средняя маржа. (Единый стиль
  KPI-карточки — переиспользуется в Аналитике.)
- **Таблица:** изображение · Лот (название + № + заказчик) · Кол-во · Макс. цена ·
  Себестоимость/шт · Срок (+ чип открыт/закрыт) · Совпадение (% + полнота: полное/
  частичное/несовпадение) · стрелка. «Загрузить ещё».
- Клик по строке → **дровер/модалка**: шапка лота (изображение, категория, №, заказчик,
  мета, кольцо совпадения); сетка **«Требования лота»** (по типам компонентов, иконки
  CPU/RAM/GPU/…, бейджи «опционально»); карточки **«Компоненты»** с выбором
  **альтернативных деталей** (radio-строки: товар, спец, цена, % совпадения, поставщик);
  правый **сайдбар-итог** (баннер совместимости, список цен по компонентам, себестоимость
  за штуку + итог, **полоска-индикатор** «использование от макс. цены», вердикт
  выгодности good/edge/bad с чистой прибылью, кнопки Сброс / Сохранить).

### 7. Аналитика (`/analytics`)

Воронка тендеров и KPI по решениям.

- **Header:** сегмент-переключатель периода **7 дней / 30 дней / 1 год** — и он
  ДОЛЖЕН реально влиять на данные (сейчас это «мёртвый» контрол), сделай его рабочим
  визуально; колокол, настройки.
- **Сетка из 5 KPI-карточек:** Просканировано лотов, Сопоставлено, Уровень принятия %,
  Выигранная стоимость (сум), Средняя уверенность % — с дельта-индикаторами
  (рост/падение к прошлому периоду).
- **Графики (единый визуальный стиль, спокойные цвета):** «Объём решений»
  (сгруппированные столбцы принято/отклонено по дням), «Распределение статусов»

## Что сохранить из текущего продукта (не терять при редизайне)

- Никаких `alert/confirm` браузера — все подтверждения через собственные диалоги.
- Тосты с действием **«Отменить»** после принять/отклонить.
- Живой статус AI-конвейера в сайдбаре (websocket) и колокол уведомлений.
- Моноширинные табличные цифры для цен/ИНН/кодов.
- Дружелюбный экран «Сервер недоступен» (можно с юмором) вместо голой ошибки.
